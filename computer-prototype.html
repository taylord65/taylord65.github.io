Design Evolution

REVISION #1: Removed radio buttons from main view.

We removed the radio buttons from the first view, replacing it by 2 buttons. One is bigger than the other since we wanted to address one of our usability goals where the user should be able to distinguish the “admin mode” (the person that will want to add “pair beacons” and associate them with “important people”) from the regular people that will attend the event and will want to locate “important people”. Attendants of the event that will want to locate “important people” will be a lot more numerous than the organizers of the event, hence we made the button “LOCATE PEOPLE AROUND ME” a lot bigger in the view, since it is more relevant.

REVISION #2: Removed “join group” authentication process

Since one of our usability goals was to begin tracking people quickly, we also considerably reduced the number of views by removing the “authentication” process. There is no need to create groups anymore since the organizer will create a local group and the app will be only useful within a range (70 meters, allowed by the beacon). The organizer (in admin mode), will pair the beacons that are part of the group and since attendants of the event are not concerned about any other group except the one that they are attendant, we decided that it wasn’t relevant having the user pick his own group, or his own event. This removes steps of access for the user and now he can just select the people that he wants to follow and then locate them, without joining a group.

REVISION #3: Changed the localization interface.

Because of feasibility reasons, we decided to not use a map for localizing the beacons (the important people) because we noticed that the beacons aren’t providing us with an orientation, but mostly with the strength of their signal. Therefore, we decided that we’ll give the user a perspective of proximity, an indication of how close are they to the person of interest. The persons of interested are now displayed vertically in order of proximity, an interface much more intuitive than one depending on circles and pin points, where It could have been confusing because the pin point might indicate an orientation within the circle.

User Population

Initial User Segments

Three target user segments were identified in the project proposal.

The festival goer

This user is mainly interested in this application for personal use, that is, to find friends and family at large events, including but not limited to festivals. In this scenario, all users participating in the interaction with our system are assumed to have smartphones. These mobile devices may however have limited functions due to network congestion.

The parent

This user will primarily use the system to find other members of their family. More precisely, this user would like to find individuals who cannot be reached by telephone.

The conference/hackathon participant

This user is interested in using our system to find and interact with individuals with whom they have no pre-existing relationship.

Refer to the project proposal for full descriptions of the intended population segments.

User Population (for test session)

As the system will be tested by other students, we expect the evaluators to identify with either (or both) the festival (large event) goer or the conference/hackathon participant category. In other words, the evaluator should have some interest in technology and should be fairly comfortable with using a smartphone. Although they do not need any particular skill, the user should have some knowledge on how to use smartphone applications. The user will be someone who has already been placed in a situation in which they had some difficulty finding a specific person. They will most likely be someone who can adapt to new situations and who is open to learning new skills.

As described in the project proposal, the user who identifies with the type festival goer will be someone who tends to spend a lot of time in social settings and who easily adapts to new situations. This ideally open-minded person is someone who is looking for ways to improve the way they interact with others. This person is mainly interested in finding people they already know.

The user who identifies with the type conference/hackathon participant may be less comfortable in large social events. This person is used to going to events where they do not know the majority of the participants. They are mainly interested in finding people they do not have a pre-existing relationship with. This person may not be actively looking for ways to find and interact with others but is open to trying a tried and tested application.

Usability Goals

Usability Goal Rationale Quantitative Target Measurement
The user should be able to associate adding a beacon with adding a person of interest. The user must know that in order for a person to be tracked, they must be holding onto a beacon that is registered in the system with their name. The user will attempt to rename a beacon after pairing it as the next step and not think that the test is complete. If the user is lost and waits idly for more than 10 seconds after pairing, the association has not been made.
Begin tracking a person of interest quickly. The app should provide the ability to produce immediate assistance. We don't want the user to become lost trying to initiate the main function of the app. The user should arrive at the screen that displays people you are tracking without deviating from the path to get there.
The app should be an effective way of locating a person of interest. The user should understand how to move in order to locate the person of interest. The user verbally confirms the distance to the selected person of interest.
The app should be straightforward and simple to use. We don't want the user getting lost or asking the examiner what to do. The user performs all activities without asking the examiner any questions.
Using the app should provide a pleasant experience. The app should not frustrate or impair the user. It needs to provide an experience that is better than wandering and searching for someone. A satisfaction rating of at least 4/6 is desirable at this point.

Usability Test Procedure

This experiment is designed to test the ease of use of our locator application. When presenting a test subject with the goal of finding a friend or person with whom they have no prior relationship with, they must be able to use our application as an assistive tool. They must also be able to add people of interest to the system so that they can become discoverable.

Locating people should be faster and less stressful than wandering and searching because of our application. People of interest are holding onto iBeacons which are detected by the application.

How many examiners are required?

One examiner is enough to oversee this test.

What Equipment will the examiners need?

Our application can be run on the iOS simulator on a mac. If you do not have a mac, contact a person in our group and we will assist in setting up a test environment. To set up the prototype, follow these steps:

  1. Download this Xcode project
  2. Extract the files from the Zip folder
  3. Double click on the file called “around_alpha.xcodeproj” to open the project file in Xcode. If you do not have Xcode on your computer, it can be downloaded from the Mac App Store.
  4. Once the project is loaded, press the play button to run the simulator.
  5. An iPhone simulator should appear.

Print off the required amount of data collection sheets for the amount of subject you will be testing with. Be prepared to fill out the sheet with test results.

How should the prototype be handled?

The test subject can sit at a desk with the computer running the simulator placed in front of him/her. The test subject can interact with the prototype by using the mouse to simulate touch events. Leave the app running on its initial screen.

What to tell the test subject?

Begin by reading the test subject the following text:

Thank you for volunteering your time to test our system. We are conducting a usability evaluation on an application that allows you to locate and meet up with people of interest in a confined space. A person of interest holds onto a beacon which emits a signal that allows them to become discoverable to other users of the application.

We will use the results of this experiment to identify problems in our system or areas that need improvement so that we can provide a better experience for the user. Before we start with the experiment, we (I) have a few short questions to ask you. After, you will be asked to perform a few simple tasks with our applications prototype. You do not need any type of special preparation for it. We (I) will tell you what each task is ahead of time but I will not tell you how to do it. When you perform the tasks, it would be very helpful if you could guide us (me) through your thought process. At the end of the experiment, we will ask you a few short questions about the system and the test.

You are strongly encouraged to be honest with your feedback; we need any and all kinds of feedback in order to improve our system. We (I) will be taking notes but keep in mind that you are not being evaluated on how well you perform the tasks. It is the system that we are evaluating and not you. If something is unclear during the test, do not hesitate to ask us (me) questions.

How should the examiner proceed with the test?

After you have read them the introduction text, ask them the pre test questionnaire questions and record their answers on the data collection sheet.

Next, read the test subject the following text to set up the usability test:

You are an organizer in charge of running a Hackathon, a coding competition for university students. Many students will need assistance in their coding projects. In order to help assist students with their projects, the Hackathon organizing committee has hired several mentors who will walk around and students can ask them questions. Students have a hard time locating mentors for help. In order to help students locate mentors more easily, mentors will be given iBeacon devices so they can be located with the app you are about to use. Bob is an Android mentor, you need to add him to the system so he can be discoverable for students. He has a Blue beacon. Your task is to assign the blue beacon to Bob. He should be discoverable with the name Android Mentor Bob. Use the application in front of you to complete this task.

Observe the test subject as he/she attempts to complete this task. The user should press “I am an admin, let me in”, click on the blue beacon, press pair beacon, and then press and hold the blue beacon to rename it to ‘Android Mentor Bob’. Pay attention to the user's actions and record observations and user comments in the data collection sheet. Also answer the questions.

When the test subject has completed this task, move on to the second task. Do not reset the application. Read them the following text:

Now imagine you are a student at the Hackathon. You are sitting at your desk and you need help on your Android project so you wish to speak with an Android Mentor. Using the application, find an Android Mentor and determine how far away he is from you.

Observe the test subject as he/she attempts to complete this task. See if they will verbally confirm how far away the person of interest is. The user should press the X in the top right to return to the home screen, then press ‘Locate People Around Me”, then select ‘Android Mentor: Maxime Gregoire”, and then press ‘Locate People’. The user should tell you he is 5 meters away.

What/How/When should the examiners measure?

As the user is attempting to complete the tasks, the examiner should be filling out the data collection sheet in the reporting section with the test results.

Usability Evaluation

Heuristics

We selected the following heuristics to evaluate our system, based on Jakob Nielsen's research.

Visibility of system status

The system requires real-time display of the distance between the user and the person of interest. As such, it should have quick and accurate visual feedback especially when locating beacons.

Match between system and the real world

The interface for adding beacons should be clear, and the interface for locating users should intuitively direct the user towards the person of interest.

User control and freedom

When a user chooses the wrong option, there should be an easy way to backtrack and try another option.

Consistency and standards

The vocabulary and visual language used in the application should be consistent accross the user experience.

Recognition rather than recall

The information the user has to remember should be minimal. The application should show all relevant information at all times.

Aesthetic and minimalist design

Only relevant information should be shown to minimize information overload, since this system is supposed to be very simple by design.
