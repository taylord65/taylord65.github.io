Usability Goals & Benchmark Tests

Script

We are currently working on developing a system that can aid with locating persons of interest in a confined setting.

Observer Briefing

Thank you for volunteering your time to test our system. We will use the results of this experiment to identify problems in our system or areas that need improvement so that we can provide a better experience for the user. Before we start with the experiment, we (I) have a few short questions to ask you (see pretest questionnaire).

You will be given a paper prototype of the system we are developing and we are (I am) going to ask you to perform a series of simple tasks. You do not need any type of special preparation for it. We (I) will tell you what each task is ahead of time but I will not tell you how to do it. When you perform the tasks, it would be very helpful if you could guide us (me) through your thought process. We (I) will simulate the behaviour of the application by changing the screen but you should pretend that we (I) am not here. At the end of the experiment, we will ask you a few short questions about the system and the test.

You are strongly encouraged to be honest with your feedback; we need any and all kinds of feedback in order to improve our system. We (I) will be taking notes but keep in mind that you are not being evaluated on how well you perform the tasks. It is the system that we are evaluating and not you. If you are okay with it, we are going to film the test. We can hide your face if you prefer. If some things are unclear during the test, do not hesitate to ask us(me) questions. If you need to take a break or pause the experiment, just let us (me) know and we will pause it.

Question to be asked during test:
If a user struggles to perform a task:

I can see that you are struggling with task X. Can you tell me why/what you are looking for?

Pre-test Questionnaire
  1. When you go to a conference, job fair, convention, hackathon, how often are you trying to locate a person of interest?
  2. How do you go about looking for this person, or how would you go about looking for this person?
  3. Do you have any experience with apps that allow you to find people of interest (find my friends, etc.)? If so, how is your experience (regular use vs used it once, helpful or not helpful)
  4. If you had to look for someone but could not call them, how would you proceed to look for them?
Test Plan #1

User: Not a person of interest, but someone who wishes to find a person of interest

Benchmark Tasks: Find a person of interest

Script to preface task for user

You are at a career fair, you see a sign on the wall that advertises a “Locator App” to find people at the career fair. (It tells you to download the app from the app store and use this access code [XXX-XXX]). You download the app and open it. You have heard of John Smith, he is the CEO of a company you are interested in applying to. You want to use the app to find him. Your task is to find John Smith in real life. The career fair is very crowded, there are people everywhere.

Question to be asked during test:
Ask the user what the triangle indicator/number means

Usability Goals:
  • Arrive at group user list screen within 2 minutes of opening the app.
  • The user presses join a group/track a beacon within 10 seconds
  • The user does not press Create a group/ Register a beacon at all
  • The user types in the correct access code on the first try
  • The user understands the triangle/distance number indicator by confirming its use after being asked.
  • The user finds the person of interest who is partially hidden 5m away within 30 seconds.
  • The user should be able to perform all activities with minimal help from external sources
  • If the user makes an error, they should be able to correct it without assistance from the facilitator
  • Satisfaction rating above 3 in post test questionnaire
Rationale for usability goals:

We want the user to quickly arrive at the screen where he can track the person of interest. The user shouldn’t be confused about the group system, and he/she should know that creating a group or registering beacons is not in the scope of the designated task. We want the user to know that the access code from the sign is used for the app, so the user should be able to enter it correctly on the first try. We want to ask the user what the triangle indicator/number means because we want to confirm that they know it indicates the distance to the person. It should be obvious and they shouldn’t be confused when asked. Finding the person who is 5m away should be an easy task so it should be completed within 30 seconds. Overall, the app should be easy to use: it should not add an unnecessary layer of complexity to the task. Most tasks should thus be learned and performed in a minimal time frame.

Test Plan #2

User: An administrator

Benchmark Tasks: Create a group, name the group, add a beacon, add a name, upload a picture of user, repeat for second kid, remove a kid, add a new kid.

Script to preface task for user

You are a dad going to the park with your 2 kids (Pedro and Pablo) and you want to keep track of them. You have the app downloaded to your phone and 2 spare ibeacons. You are at the park and you sit down at a table, you place the ibeacons in front of you and open up the app on your phone. Your two kids are in front of you. You want to set it up so you can track your kids at anytime. What do you do?

[once second kid has been set up]
Pablo gives his beacon to you and he goes home. Pedro runs into his friend Ryan at the park, you need to supervise Pedro and Ryan, you want to add Ryan to your group to be able to locate him as well. How do you do this?

Usability Goals:
  • The user presses Create Group/ Register beacon within 10 seconds
  • The user does not press Join a group/track a beacon at all
  • The user adds the first child by clicking a beacon id, renaming it to Pedro or Pablo,
  • The user takes a picture of the child they just entered.
  • The user repeats the process with no errors for the other child
  • The user removes Pablo
  • The user presses + Add a new user
  • The user repeats the process for Ryan with no errors.
  • The user should be able to perform all activities with minimal help from external sources
  • If the user makes an error, they should be able to correct it without assistance from the facilitator
  • Satisfaction rating above 3 in post test questionnaire
Rationale for usability goals:

We expect the user to know that he is the administrator and that creating a group/registering a beacon is the only option for his role, thus it should be completed quickly. The system should easily guide the user through the setup process so the user should be able to meet all the goals of adding the kids. All these incremental steps should be accomplished in succession without errors. Again, because the app is a tool to help users perform a task, it should be easy to learn and use and should not add an unnecessary layer of complexity to the task.

Post-test Questionnaire
  1. Generally, how do you find your encounters with the system, on a scale of 1 to 6, where 1 is not useful at all and 6 is very useful?
  2. What are your suggestions for improving the system? Is there a feature you wished you had access to?
  3. What did you like the most about the system?
  4. What did you like the least about the system?

Conclusions

The subject pressed “Create a group” instead of the intended “Join a group”. This led him down the wrong path and he kept getting more and more confused as nothing in this user flow makes sense for tracking a person. The subject was not able to figure out how to reset the application and return to the home screen. The test had to be ended and we tried again where we told him to try the “Join a group” option. He was then able to enter the access code and view the people to track.

Suggested improvements

Removing the feature of groups or at least giving more context to the group system are options. A clearer way to reset the system and return to home should be included to avoid the user from becoming lost. The indicator for closeness to a person has too many things trying to describe it. The word distance is above signal strength icons. The user understands these two concepts are linked but he has to make that association on his own. The current indicator is not completely obvious as to what it does.

Summary

Usability Goals

Effective: effective to use
Can the user use the system to do the work he needs to do?

Efficient: efficient to use
How much time it takes to use the interface and how likely it is for the user to make mistakes?

Utility: have good utility
Does the system provide all the functionality that user needs?

Learnable: easy to learn
Can users figure out what to do by exploring the interface?

Memorable: easy to remember how to use
How easy is it to remember how to use an interface after the user has experience with the system?

Safe: safe to use
What kind of errors can users make and how can they recover from the mistake?

Conclusions
  • The user was initially confused by the radio buttons
  • Poor efficiency: the interface is confusing for the user, he doesn’t know if he should pick 2 beacons at the time or one
  • The user made mistakes (e.g. choosing two beacons at a time); the facilitator had to intervene
  • The user was able to find the person of interest quickly and had no trouble understanding
  • The use of numbers indicating the distance to the person of interest was easy to understand for the user.
  • Overall however, the user was able to learn the app in a minimal time frame
Suggestions
  • Make clear that we are using radio-buttons and not checkboxes
  • A more complex prototype will not allow the user to select two beacons at a time. Alternatively a short text indicating that ‘only one beacon can be paired at a time’ could be displayed. The remaining options could also be pushed to the background, or faded out.
